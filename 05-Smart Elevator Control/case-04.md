# Case Study: Database Design for Smart Elevator Control Platform


## Problem Statement

A fast-growing infrastructure technology company, LiftGrid Systems, builds intelligent elevator control platforms for large commercial buildings across India.

Their software is used in corporate towers, malls, airports, hospitals, and high-rise residential complexes where multiple elevators operate together across many floors.

Unlike small standalone lifts, these buildings run multiple elevators per building, often grouped into zones, handling thousands of passengers daily. The system must efficiently manage elevator assignments, floor requests, maintenance tracking, and ride logs.

Each building can contain multiple elevator shafts, and each shaft contains one elevator. Elevators move across a defined set of floors and respond to ride requests generated by users from different floors.

The system should support:

* Multiple buildings
* Multiple elevators within each building
* Floor-level request tracking
* Ride allocation to elevators
* Elevator status monitoring
* Maintenance tracking
* Usage history logging

This backend platform helps operations teams monitor performance and ensures elevators remain safe, efficient, and available.

## Objective

Design an ER diagram for this smart elevator control system.

This is a multi-building infrastructure management system handling real-time movement requests, elevator allocation, and operational tracking.

## Functional Expectations

Your design should support answering the following:

* How many buildings are connected to the platform?
* How many elevators exist inside a building?
* Which floors belong to which building?
* Which elevator serves which floors?
* What requests were generated from which floors?
* Which elevator handled a request?
* Can multiple elevators serve the same floor?
* Can one elevator serve multiple floors?
* What is the status of each elevator (idle, moving, maintenance)?
* How many rides did an elevator complete today?
* Which elevator handled the most requests?
* Which requests are still pending?
* Can an elevator be temporarily disabled for maintenance?
* Can maintenance history be tracked per elevator?
* Can ride logs be recorded for analytics later?

## Scope of ER Diagram

Your ER diagram should include:

* Buildings
* Floors
* Elevators
* Elevator shafts (optional but recommended)
* Floor requests
* Ride assignments
* Ride logs / trips
* Elevator status tracking
* Maintenance tracking
* Primary keys and foreign keys
* Relationships and cardinality

## Design Considerations

* One building contains multiple floors.
* One building contains multiple elevators.
* Elevators belong to buildings.
* Elevators serve multiple floors.
* Floors can be served by multiple elevators.
* Users generate floor requests.
* Requests are assigned to elevators.
* One request results in one ride allocation.
* Elevators complete multiple rides daily.
* Maintenance history should be preserved and not overwritten.
* Status tracking should be separate from configuration data.
* Avoid storing dynamic ride data directly within the elevator entity.
